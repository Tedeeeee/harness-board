## Problem Statement

별소프트는 현재 공식 온라인 존재감이 전무하다. 홈페이지, SNS, 블로그, 회사소개서 등 외부에 회사를 알리는 채널이 하나도 없는 상태다. 잠재 고객이나 파트너가 별소프트를 검색하면 아무것도 나오지 않는다. 이 홈페이지는 별소프트의 첫 번째 외부 접점이 되어야 한다.

동시에 이 프로젝트는 FDE 과정의 결과물이기도 하다. 자유게시판 CRUD(생성/조회/수정/삭제), 검색, 페이지네이션은 과정 필수 구현 요건이다.

## Demand Evidence

- 외부에서 검증된 수요 없음. 고객이 홈페이지를 요청하거나, 홈페이지 부재로 불이익을 받은 구체적 사례 없음.
- 내부 판단에 기반: "회사 홈페이지가 있어야 한다"는 내부적 필요.
- 현재 회사 소개 채널이 전무하다는 것 자체가 갭(gap). 하지만 이 갭의 시급성은 검증되지 않았다.
- FDE 과정 요구사항이 구현의 직접적 동기.

## Status Quo

현재 아무것도 없다. PPT, PDF, 블로그, SNS — 어떤 형태의 회사 소개 자료도 존재하지 않는다. 이 홈페이지가 별소프트의 첫 번째 공식 자료가 된다.

## Target User & Narrowest Wedge

**대상 사용자:** 별소프트와 거래하거나 협업할 수 있는 잠재 고객/파트너. 별소프트를 검색하거나, 누군가에게 별소프트를 소개받아 확인하러 오는 사람.

**Narrowest Wedge:** About 페이지 하나. 회사가 무엇을 하는 곳인지 알 수 있고, 연락할 수 있는 이메일/전화가 보이면 MVP의 핵심 가치는 달성된다. 자유게시판은 FDE 과정 요건으로 추가된다.

## Constraints

- FDE 과정 필수 요건: 자유게시판 CRUD + 검색 + 페이지네이션 반드시 구현
- 배포 불필요: 로컬 환경에서 실행하여 결과물 확인
- 프론트엔드/백엔드 분리 아키텍처 선택
- 회원 시스템 없음, 댓글 없음, 관리자 패널 없음
- 회사 소개 외 추가 페이지(채용, 포트폴리오 등) 없음
- 게시글 수정/삭제 인증: 회원 시스템이 없으므로 **게시글 작성 시 비밀번호(4자리 이상)를 입력**하고, 수정/삭제 시 해당 비밀번호를 확인한다. 비밀번호는 bcryptjs로 해시하여 저장한다.
- About 콘텐츠: 더미 데이터로 구현하고, 실제 데이터는 출시 전 교체한다.

## Premises

1. 별소프트는 공식 온라인 존재감이 전무하고, 이 홈페이지가 첫 번째 외부 접점이다.
2. 홈페이지의 핵심 목적은 잠재 고객/파트너에게 신뢰감을 주고 연락으로 연결하는 것이다.
3. 자유게시판은 FDE 과정 필수 구현 요건으로 유지한다. 별도 전환 계획 없음. About 페이지에 이메일/전화 연락처를 명확히 노출하여 실제 비즈니스 연락 경로를 확보한다.

## Cross-Model Perspective

독립 AI 서브에이전트가 이 세션의 대화를 보지 않은 상태에서 검토한 결과:

**Steelman:** 별소프트가 만드는 건 "신뢰의 닻(credibility anchor)" — 최소한이지만 세련된 기업 사이트로 잠재 고객/파트너가 회사의 존재를 확인하고, 무엇을 하는 곳인지 이해하고, 연락할 수 있게 해주는 것.

**핵심 통찰:** "FDE 과정 요구사항"이라는 답변이 가장 중요. 요구사항이 사용자가 아닌 평가자를 위해 작성되었다는 뜻. 게시판 스펙은 풀스택 역량 증명에 최적화되어 있지 잠재 고객 서비스를 위한 것이 아니다. 이 두 목표를 분리하지 않으면 사이트가 학생 프로젝트처럼 보일 것.

**도전된 전제:** "향후 문의게시판으로 전환 가능한 구조 고려"는 불필요. 실제 문의는 이메일/전화로 올 것이고, 게시판으로 오지 않을 것. → 전제 수정 반영 완료.

**48시간 프로토타입 제안:** Next.js + Tailwind + SQLite/Prisma, Vercel 배포 제안. (채택되지 않음 — 배포 불필요 + 프론트/백 분리를 위해 Approach B 선택.) "게시판은 과정을 통과하기 위해 만들고, 나머지 사이트는 고객을 잡기 위해 만든다."

## Approaches Considered

### Approach A: Next.js Full-Stack (Minimal Viable)
- **Stack:** Next.js 15 (App Router) + Tailwind CSS + SQLite + Prisma
- **Deploy:** Vercel (git push = auto deploy)
- **Effort:** S~M | **Risk:** Low
- **Pros:** 한 프레임워크로 통합, Server Actions으로 CRUD 간편, 즉시 배포 가능
- **Cons:** SQLite/Vercel serverless 궁합 이슈, FDE 과정이 분리 구조를 요구할 수 있음
- **Reuses:** N/A (신규 프로젝트)

### Approach B: Frontend/Backend 분리 (Ideal Architecture) — 선택됨
- **Stack:** React (Vite) + Tailwind CSS | Express + SQLite + Prisma
- **Deploy:** 로컬 실행만 (배포 불필요)
- **Effort:** M | **Risk:** Low (배포 제외로 복잡도 감소)
- **Pros:** RESTful API 명확한 분리, 풀스택 역량 증명에 유리, 각 레이어 독립 개발 가능
- **Cons:** 프론트/백 동시 실행 필요, CORS 설정 필요
- **Reuses:** N/A (신규 프로젝트)

### Approach C: Classic SSR (Course-Aligned)
- **Stack:** Express + EJS + SQLite + vanilla CSS
- **Deploy:** 로컬 실행
- **Effort:** S | **Risk:** Low
- **Pros:** 가장 단순, 클래식 CRUD 패턴, 학습/이해 용이
- **Cons:** React 경험 없음, 디자인 품질 한계, 포트폴리오 인상 약함

## Recommended Approach

**Approach B: Frontend/Backend 분리.**

이유: 배포가 불필요해지면서 B의 최대 단점(배포 포인트 2개)이 사라진다. React + Express 분리 구조는 FDE 과정 평가에서 풀스택 역량을 가장 명확하게 보여주고, RESTful API 설계 경험도 남긴다. Tailwind CSS로 About 페이지의 디자인 품질을 확보하여 "학생 프로젝트"가 아닌 실제 기업 사이트의 인상을 줄 수 있다.

**스택 상세:**
- Frontend: React 18+ (Vite) + React Router + Tailwind CSS + Axios
- Backend: Express + Prisma ORM + SQLite
- 개발 환경: 프론트 `npm run dev` (Vite, port 5173) + 백엔드 `npm run dev` (nodemon, port 3000)

**와이어프레임:** `docs/designs/wireframe-sketch.html` 참조

**핵심 레이아웃 구조:**
```
┌─────────────────────────────────────┐
│ [로고: 별소프트]     [About] [Board] │  ← 공통 헤더
├─────────────────────────────────────┤
│                                     │
│  About: 히어로 문구                  │
│  → 3개 가치 카드 (그리드)            │
│  → 회사 정보 테이블                  │
│  → 연락 CTA (이메일/전화 버튼)       │
│                                     │

│  → [번호|제목|작성자|조회|작성일]     │
│  → 페이지네이션 (1, 2, 3...)        │
```

## Data Model

```
Post {
  id        Int       @id @default(autoincrement())
  title     String                          // 제목 (필수, 최대 100자)
  content   String                          // 본문 (필수)
  author    String                          // 작성자명 (필수, 자유 입력 문자열, 최대 20자)
  password  String                          // 비밀번호 (필수, bcryptjs 해시, 수정/삭제 시 확인용)
  views     Int       @default(0)           // 조회수
  createdAt DateTime  @default(now())       // 작성일
  updatedAt DateTime  @updatedAt            // 수정일
  deletedAt DateTime?                       // 삭제일 (soft delete, null이면 미삭제)
}
```

## API Spec

| Method | Endpoint | Description | Request Body | Response |
|--------|----------|-------------|-------------|----------|
| GET | `/api/posts` | 게시글 목록 (페이지네이션+검색) | Query: `page=1&size=10&searchType=title&keyword=` | `{ posts: [...], totalCount, totalPages, currentPage }` |
| GET | `/api/posts/:id` | 게시글 상세 (조회수 +1) | - | `{ id, title, content, author, views, createdAt }` |
| POST | `/api/posts` | 게시글 작성 | `{ title, content, author, password }` | `{ id }` (201) |
| PUT | `/api/posts/:id` | 게시글 수정 | `{ title, content, password }` | `{ id }` (200). 수정 시 `author`는 변경 불가, 원본 유지. |
| DELETE | `/api/posts/:id` | 게시글 삭제 (soft delete) | `{ password }` | (204) |

**에러 응답:**
- `400` — 필수 항목 누락 또는 유효성 실패
- `401` — 비밀번호 불일치 (수정/삭제 시)
- `404` — 게시글 없음 또는 삭제됨

**검색:** 필드 선택 드롭다운(`searchType`: title / content / author) + 검색어 입력창. 선택한 필드에 대해 LIKE 검색.

**페이지네이션:** offset 기반, 페이지당 10개, 번호 클릭 방식 (1, 2, 3...).

## Success Criteria

- About 화면에서 회사명, 주요 사업 설명, 연락처(이메일/전화)가 스크롤 없이 첫 화면(above the fold)에 모두 노출된다
- 연락처 CTA(이메일 문의 버튼, 전화 연결 버튼)가 About 화면에 존재한다
- 자유게시판 CRUD 전체 흐름(목록→작성→상세→수정→삭제)이 동작한다
- 게시글 작성 시 비밀번호 입력, 수정/삭제 시 비밀번호 확인이 동작한다
- 게시판 검색(필드 선택 드롭다운 + 검색어 입력)이 동작한다
- 게시판 페이지네이션(페이지당 10개, 번호 클릭 방식)이 동작한다
- 예외 상황(게시글 없음, 검색결과 없음, 삭제된 글 접근, 비밀번호 불일치)이 처리된다
- 프론트엔드/백엔드 분리 구조가 명확하다 (RESTful API, CORS 설정)

## Dependencies

- Node.js 18+ 설치
- **Frontend 패키지:** react, vite, tailwindcss, react-router-dom, axios
- **Backend 패키지:** express, prisma, @prisma/client, cors, bcryptjs (비밀번호 해싱, 순수 JS — Windows에서 네이티브 빌드 불필요)
- **개발 환경:** 터미널 2개로 프론트(port 5173)/백(port 3000) 동시 실행. 또는 루트 `package.json`에 `concurrently` 사용하여 `npm run dev` 한 번으로 양쪽 실행.
- FDE 과정 제출 기준 확인 (스택 제약 여부)

## The Assignment

**이번 주 안에 FDE 과정 담당자/평가 기준을 확인하세요.** 특히:
1. 기술 스택 제약이 있는지 (React/Express가 허용되는지)
2. 반응형 구현이 평가 항목에 포함되는지

이 2가지가 확인되어야 구현에 안정적으로 들어갈 수 있습니다. 확인 없이 구현을 시작하면, 과정 요구사항과 어긋나서 다시 만들어야 할 위험이 있습니다.

## What I noticed about how you think

- 자유게시판 유지에 대해 내가 두 번 도전했을 때, 처음에는 "게시판 유지"를 선택하고, 이유를 물었을 때 "FDE 과정 요구사항"이라고 명확하게 답했습니다. 감정적 방어가 아니라 실용적 제약으로 판단한 것이 인상적입니다.
- 세컨드 오피니언이 전제를 도전했을 때 "전제 수정"을 선택했습니다. 좋은 논거가 나오면 원래 계획을 바꿀 수 있다는 것 — 유연함과 실용성이 보입니다.
- 접근법 선택에서 가장 쉬운 것(A)이 아니라 아키텍처가 명확한 것(B)을 골랐고, 동시에 "배포는 안 해도 돼"라고 덧붙였습니다. 복잡해 보이는 것을 고르되 불필요한 부분은 빼는 — 실무적 판단입니다.
