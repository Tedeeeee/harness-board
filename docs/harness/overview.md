# Harness Overview

## Why This Exists

일반적인 AI 코딩 환경은 "질문하면 답하고, 시키면 구현하는" 수준에서 머무르기 쉽습니다.
Harness Board는 그보다 한 단계 더 나아가, 요구사항 정제부터 계획, 구현, 검증, 출시 준비까지 이어지는 개발 사이클을 하나의 일관된 운영 체계로 만들기 위해 만들어졌습니다.

핵심 목표는 다음과 같습니다.

- 요구사항 문서를 실제 실행 가능한 개발 흐름으로 변환한다
- 코딩 이전에 요구사항과 계획을 먼저 정리한다
- 구현보다 검증과 안전장치를 우선하는 습관을 강제한다
- 사람이 모든 단계를 직접 지시하지 않아도, Claude가 구조적으로 움직일 수 있게 한다
- 외부 설치물 없이 이 레포 하나만으로 재현 가능한 하네스를 유지한다

## Core Idea

이 하네스는 단순한 프롬프트 모음이 아닙니다.
Claude가 프로젝트 안에서 일관된 방식으로 행동하도록 만드는 운영 레이어에 가깝습니다.

구성은 크게 네 가지입니다.

1. Slash commands
   - Claude에서 직접 호출하는 진입점입니다.
   - `/office-hours`, `/blueprint`, `/review`, `/ship` 같은 명령이 여기에 해당합니다.

2. Skills
   - 각 커맨드가 실제로 따라야 할 상세 행동 지침입니다.
   - 요구사항 정제, 계획 수립, 코드 리뷰, 출시 같은 반복 가능한 작업 절차를 정의합니다.

3. Agents
   - `planner`, `architect`, `code-reviewer` 같은 역할 중심 정의입니다.
   - 복잡한 작업을 역할 단위로 분리해 사고할 수 있도록 돕습니다.

4. Hooks
   - Claude가 작업 중 위험한 행동을 하거나 품질을 해칠 가능성이 있을 때 이를 제어하는 자동 안전장치입니다.

## Design Principles

### 1. Requirements First

코드보다 먼저 요구사항을 정리합니다.
무엇을 만들지 불분명한 상태에서 구현으로 들어가지 않습니다.

### 2. Plan Before Build

좋은 구현은 좋은 계획에서 시작합니다.
계획 없는 자동화는 속도는 빠를 수 있어도 방향이 흔들리기 쉽습니다.

### 3. Safe Autonomy

Claude가 자율적으로 움직이더라도, 위험한 수정이나 품질 저하는 제어되어야 합니다.
Hooks와 명확한 역할 분리는 이를 위한 최소 안전장치입니다.

### 4. Minimal Human Intervention

사람은 모든 단계를 직접 조작하지 않고, 중요한 순간에만 결정합니다.
이 하네스는 그 개입 지점을 최소화하는 방향으로 설계됩니다.

### 5. Standalone First

외부 개인 환경, 절대 경로, 특정 플러그인 설치에 의존하지 않습니다.
누가 받아도 같은 구조를 이해하고 실행할 수 있어야 합니다.

## Who This Is For

이 하네스는 다음과 같은 경우에 특히 잘 맞습니다.

- 요구사항 문서 기반으로 개발을 시작하는 팀
- Claude Code를 단순 도우미가 아니라 실행 파트너로 쓰고 싶은 경우
- AI가 마음대로 코드를 쓰는 것이 아니라, 구조 안에서 움직이게 만들고 싶은 경우
- 반복 가능한 개발 프로세스를 프로젝트 단위로 내재화하고 싶은 경우

## Long-Term Vision

Harness Board의 최종 목표는 "Claude가 알아서 해주는 환경"이 아닙니다.
그보다 더 중요한 목표는, Claude가 어떤 기준과 절차 위에서 움직여야 하는지를 명확히 하는 것입니다.

좋은 하네스는 많은 기능을 붙이는 것이 아니라,

- 진입점이 명확하고
- 역할이 분리되어 있으며
- 위험이 통제되고
- 문서와 실제 동작이 일치하는 상태

를 유지하는 데서 시작합니다.

Harness Board는 바로 그 기반을 만드는 프로젝트입니다.
